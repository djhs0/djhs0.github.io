        go中隐式间接引用的疑惑

        djhs0 2019-10-15 interface pointer

        # 隐式间接引用

        在学习go语言时，跟着go语言之旅学习了go的语法和其中的概念，对结构体指针有如下的描述

        结构体指针 结构体字段可以通过结构体指针来访问。

        如果我们有一个指向结构体的指针 p，那么可以通过 (*p).X 来访问其字段 X。不过这么写太啰嗦了，所以语言也允许我们使用隐式间接引用，直接写 p.X 就可以 这样使用的时候确实方便了很多，但是如果理解不清楚的话，在使用中还是会出现令人莫名其妙的情况。

        # 接口实现中的隐式间接引用

        type Inter interface {
	Method()
}

type A struct{}

func (a *A) Method() { fmt.Println("A impl") }

type B struct{}

func (b B) Method() { fmt.Println("B impl") }

func Execute(in Inter) {
	in.Method()
}

func main() {
	a := A{}
	Execute(a)  // A does not implement Inter (Method method has pointer receiver)
	Execute(&a)  //正常

	b := B{}
	Execute(b)  //正常
	Execute(&b)  //正常
}

        在上面代码中，A对接口进行了指针实现，B对接口进行了值实现，但是在使用中两者出现了不同，值a并不认为实现了Inter接口，对于&b，编译器却认为实现了Inter接口。
        也就是说，在编译过程中：

        1. 编译器没法将值a转换成指针类型&a
        2. 编译器可以将指针类型&b转换成值类型b

        接口在项目中使用的很频繁，如果搞不清楚上述动作的原因，在使用中经过多个接口之间的转换之后就会发晕。

        # 接口转换现象的原因

        在go中什么时候需要将方法定义到结构体指针上？
        当在方法中需要修改结构体的内容时，当方法定义在结构体的值上时不能修改结构体的内容。

        这样就形成了一个单向的转换

        1. 当方法是定义在结构体值上时，指针可以当作值来使用，因为编译器已经确定方法不会改变结构体中的内容.
        2. 当方法定义在指针上时，值并不能当作指针来使用，因为编译器不能确定在指针方法中有没有改变结构体中的内容。
