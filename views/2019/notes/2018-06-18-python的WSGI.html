        # WSGI

        1. WSGI和uwsgi 都是协议，而不是服务器。它用于与Web服务器进行通信以实现负载平衡，特别是利用纯HTTP无法提供的额外功能。到目前为止，Nginx和Cherokee已经实现了这个协议。
        2. uWSGI是一个服务器，它实现的协议之一是WSGI（不要将uwsgi协议与uWSGI服务器混淆）。WSGI是一个Python 规范。有几种WSGI规范的实现，它的目的不仅仅是用于应用程序服务器/ Web服务器，而且还有很多WSGI应用程序服务器（即CherryPy，它也恰好有一个生产就绪WSGI兼容的Web服务器，如果你已经不够困惑了！）。
        3. 比较uwsgi和WSGI是比较桔子和苹果。

        WSGIserver收到http请求之后根据WSGI规范，调用django框架实现的WSGI接口WSGIHandler,这个WSGIHandler实现了__call__方法，加载中间件，然后返回一个可调用对象application，然后application被服务器通过application(environ, start_response)的形式来做，所有的操作都是在这个函数中完成。

        # WSGI

        python web server gateway interface. 它规范了web server如果与 web application通信，web applications怎样链接在一起去处理一个请求。
        这个描述定义在PEP3333中。

        WSGI中定义了三个角色web服务器，中间件，web应用

        # 应用程序

        一个应用程序对象简单来讲就是一个接收2个参数的可调用对象。django中是一个实现了__call__方法的类的实例。返回一个可迭代对象。

        def app(environ, start_response):
    status = '200 OK'
    response_headers = [('content-type', 'text/plain')]
    start_response(status, response_headers)
    return [b'hello world!']

        # web服务器

        当http客户端向web应用程序发来一个请求，web服务器都会去调用应用程序提供的可调用对象。 web服务器会传给web应用调用对象一个environ和自己定义的start_response可调用对象，start_response接收两个必要参数，status, response_headers,和一个可选参数exc_info。

        # 中间件

        中间件可扮演两端角色，对于服务器来讲它可以是应用程序，对于它包含的应用程序来说，它可以被看做服务器。
        可以实现的功能

        • 重写环境变量，根据目标 URL，将请求消息路由到不同的应用对象。
        • 允许在一个进程中同时运行多个应用程序或应用框架。
        • 负载均衡和远程处理，通过在网络上转发请求和响应消息。
        • 进行内容后处理，例如应用 XSLT 样式表。

    """创建WSGI中间件，对USER_AGENT进行检测
    """
    def __init__(self, application):
        self.application = application

    def __call__(self, environ, start_response):
        if 'Mozilla/5.0' not in environ.get('HTTP_USER_AGENT'):
            status = '111 HAHA'
            response_headers = [('content-type', 'text/plain')]
            start_response(status, response_headers)
            return [b'You are not Mozilla/5.0']
        return self.application(environ, start_response)
