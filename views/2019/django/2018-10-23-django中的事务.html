      # django数据库事务

      在数据库系统中，一个事务是指一系列数据库操作组成的一个完成的逻辑过程。比如一个银行转账过程，从原账户扣除金额，往目标账户中添加金额，这个两个操作组成了一个数据库事务。

      # 满足特性

      一个数据库事务需要满足以下4个特性

      1. 原子性
        指一个事务所包含的所有操作要么全部成功，要么全部失败。
      2. 一致性
        指事务在事务开始之前和事务结束之后，数据库的完整性没有遭到破坏。这表示写入的资料必须完全符合预期，这包括资料的精确性，串联性以及后续数据库能够自发性的完成预定工作。(比如转账之前A账户300，B账户200，综合500。转账之后不能总和变成了不是500了。)
      3. 隔离性
        数据库允许多个并发事务同时对其数据进行读取和修改，隔离性可以防止多个事务并发执行时由于交叉执行而导致数据的不一致。
      4. 持久性
        指事务结束后，对数据库的修改是永久性的，即使系统故障也不会丢失。

      # Django中的事务

      # 默认数据库事务

      django中ORM中的事务是在一次查询范围上的。

      Each query is immediately committed to the database, unless a transaction is active.

      所以在使用ORM执行save或者update操作时，每一个操作都是一个独立的事务，save之后就立马commit了。

      # 配置数据库事务

      # 1.在一个事务中包含大于一个的数据库操作

      开启数据库事务可以使用

      from django.db import transaction

      1. (无效)使用transaction.atomic装饰器
      @transaction.atomic():
def register(request):
    try:
        user = request.user
        day = request.POST.get('day')
        user.coin_nums -= int(day)
        trade = TradeRecord()
        trade.user = user
        trade.amount = int(day)
        trade.time = int(day)
        user.save()
        trade.save()

    except ObjectDoesNotExist:
        return JsonResponse({"res": "error！"})
    return JsonResponse({"res": "OK！"})

      再上例中使用了在view函数上加入装饰器的方式在view中开启事务，在view成功完成时自动commit事务。注意：上例中的数据库操作出现的异常会被view中定义的try-except捕获，transaction.atomic捕获不到异常就不会去rollback，所以无论如何view都会执行成功，最后都会commit，这样就违背初衷了。
      在1中如果去掉try-except并不是个好的方式，删除try会导致异常被抛出调用链，并且在其他地方出现。所以使用以下方式是一种解决办法。 2. (有效)使用with管理事务

      def register(request):
    try:
        with transaction.atomic():
            user = request.user
            day = request.POST.get('day')
            user.coin_nums -= int(day)
            trade = TradeRecord()
            trade.user = user
            trade.amount = int(day)
            trade.time = int(day)
            user.save()
            trade.save()

    except ObjectDoesNotExist:
        return JsonResponse({"res": "error！"})
    return JsonResponse({"res": "OK！"})

      # 2.对每个HTTP请求启用事务

      django中允许你对每个http请求都启用transaction。在这种情况下，django会自动装饰每个view函数，加入视图函数抛出异常，django就会回滚事务。
      配置方式，在settings.py文件中

      DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': os.path.join(SITE_ROOT, 'test.db'),
        'ATOMIC_REQUEST': True,  # 启用Transaction per request mode
    }
}

      # 3.设置savepoint

      使用with transaction.atomic()后再捕获异常之后会调用rollback，然后回滚到transaction.atomic()开始的时候。同样的也可以设置savepoint。
      savepoint设置在一个transaction中，设置savepoint可以使用户只回滚或者提交一部分事务。支持savepoint的数据库为SQLite (≥ 3.6.8), PostgreSQL, Oracle 和 MySQL (当使用 InnoDB storage engine)
      设置savepoint主要使用

      1. savepoint(using=None)
      2. savepoint_commit(sid, using=None)
      3. savepoint_rollback(sid, using=None) 三个函数，具体使用方法
      from django.db import transaction

# open a transaction
@transaction.atomic
def viewfunc(request):
    a.save()
    # transaction now contains a.save()
    
    sid = transaction.savepoint()
    b.save()
    # transaction now contains a.save() and b.save()
    if want_to_keep_b:
        transaction.savepoint_commit(sid)
        # open transaction still contains a.save() and b.save()
    else:
        transaction.savepoint_rollback(sid)
        # open transaction now contains only a.save()

      1. get_rollback(using=None)
      2. set_rollback(rollback, using=None)

      # 4.嵌套事务

      实际上不适用savepoint也可以实现部分回滚的效果。可以使用嵌套事务来实现。

      def save_points(self,save=True):
    with transaction.atomic():
        user = request.user
        day = request.POST.get('day')
        user.coin_nums -= int(day)
        with transaction.atomic()：
            trade = TradeRecord()
            trade.user = user
            trade.amount = int(day)
            trade.time = int(day)
            trade.save()
        user.save()
